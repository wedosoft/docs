# 인시던트 분류 및 우선순위

인시던트의 정확한 분류와 우선순위 설정은 한정된 리소스를 효율적으로 활용하고 비즈니스 영향을 최소화하는 핵심 프로세스입니다.

:::info 인시던트 분류 체계 구축 전 필수 확인사항
- **비즈니스 우선순위 정의**: 조직의 핵심 비즈니스 프로세스와 시스템 중요도 사전 합의 필요
- **SLA 정책 연계**: 고객 SLA와 내부 OLA(Operational Level Agreement) 기준 연동 필수
- **에스컬레이션 매트릭스**: 우선순위별 대응 시간과 에스컬레이션 경로 사전 정의 필요
- **업무 시간 고려**: 24/7 서비스와 업무시간 서비스의 우선순위 차등 적용 검토
:::

## 인시던트 분류 체계

### 1. 서비스 기반 분류

비즈니스 서비스 단위로 인시던트를 분류하여 영향도를 명확히 파악합니다.

**핵심 서비스 카테고리**:

| 서비스 등급 | 서비스 유형 | 비즈니스 영향 | 대응 시간 |
|-------------|-------------|---------------|-----------|
| **Tier 1 (Critical)** | 고객 대면 서비스, 매출 직결 | 즉시 매출 손실 | 15분 이내 |
| **Tier 2 (Important)** | 내부 핵심 업무 시스템 | 업무 중단 | 1시간 이내 |
| **Tier 3 (Standard)** | 일반 업무 지원 도구 | 생산성 저하 | 4시간 이내 |
| **Tier 4 (Low)** | 개발/테스트 환경 | 제한적 영향 | 24시간 이내 |

**서비스별 세부 분류**:
```yaml
고객 서비스 (Tier 1):
  - 전자상거래 플랫폼
  - 고객 포털/앱
  - 결제 시스템
  - 고객 지원 시스템

핵심 업무 시스템 (Tier 2):
  - ERP 시스템
  - CRM 시스템
  - 인사/급여 시스템
  - 회계/재무 시스템

지원 시스템 (Tier 3):
  - 이메일 시스템
  - 파일 서버
  - 협업 도구
  - 프린터/스캐너

개발 환경 (Tier 4):
  - 개발 서버
  - 테스트 환경
  - 스테이징 환경
```

### 2. 기술적 분류

기술적 관점에서의 체계적 분류를 통해 전문 팀 배정과 해결 전략을 수립합니다.

**인프라 계층별 분류**:

1. **네트워크 (Network)**
   - WAN/LAN 연결 장애
   - 라우터/스위치 오류
   - DNS/DHCP 문제
   - 방화벽 및 보안 장비

2. **서버 인프라 (Server)**
   - 물리 서버 하드웨어
   - 가상화 플랫폼 (VMware, Hyper-V)
   - 운영체제 (Windows, Linux)
   - 스토리지 시스템

3. **애플리케이션 (Application)**
   - 웹 애플리케이션
   - 데이터베이스 시스템
   - 미들웨어
   - API 서비스

4. **클라이언트 (End-User)**
   - 데스크톱/노트북
   - 모바일 기기
   - 소프트웨어 라이센스
   - 프린터/주변기기

:::tip 효과적인 분류를 위한 모범사례
**다면적 분류 접근법**:
- 서비스 영향도와 기술적 카테고리를 조합하여 정확한 분류 실현
- 자동 분류 규칙과 수동 검토를 병행하여 분류 정확도 향상
- 정기적인 분류 체계 검토를 통해 비즈니스 변화 반영
:::

## 우선순위 매트릭스

### ITIL 기반 우선순위 결정

영향도(Impact)와 긴급도(Urgency)를 조합한 표준 매트릭스를 사용합니다.

**우선순위 매트릭스**:

|                | **높은 긴급도** | **중간 긴급도** | **낮은 긴급도** |
|----------------|----------------|----------------|----------------|
| **높은 영향도** | P1 (Critical) | P2 (High) | P3 (Medium) |
| **중간 영향도** | P2 (High) | P3 (Medium) | P4 (Low) |
| **낮은 영향도** | P3 (Medium) | P4 (Low) | P4 (Low) |

### 영향도 평가 기준

**높은 영향도 (High Impact)**:
- 50명 이상 사용자 영향
- 핵심 비즈니스 프로세스 중단
- 고객 서비스 완전 중단
- 데이터 손실 위험
- 보안 침해 사고

**중간 영향도 (Medium Impact)**:
- 10-50명 사용자 영향
- 일부 비즈니스 기능 제한
- 성능 현저히 저하
- 워크어라운드 존재하나 복잡

**낮은 영향도 (Low Impact)**:
- 10명 미만 사용자 영향
- 개별 사용자 문제
- 성능 경미한 저하
- 쉬운 워크어라운드 존재

### 긴급도 평가 기준

**높은 긴급도 (High Urgency)**:
- 업무 시간 내 핵심 업무 중단
- 고객 SLA 위반 임박
- 규정 준수 이슈
- 미디어 노출 우려

**중간 긴급도 (Medium Urgency)**:
- 업무에 영향 있으나 대체방안 존재
- SLA 여유시간 충분
- 계획된 작업 지연 가능

**낮은 긴급도 (Low Urgency)**:
- 업무 시간 외 발생
- 즉시 해결 불필요
- 다음 정기 점검시 처리 가능

:::warning 우선순위 설정 시 주의사항
- **비즈니스 시간 고려**: 업무 시간 외에는 긴급도 재평가 필요
- **VIP 사용자 가중치**: 임원진이나 핵심 고객의 이슈는 우선순위 상향 고려
- **연쇄 효과 분석**: 작은 문제가 큰 시스템 장애로 확산될 가능성 평가
:::

## 비즈니스 영향 평가

### 정량적 영향 평가

**매출 영향 계산**:
```markdown
시간당 매출 손실 = 일 평균 매출 ÷ 24시간
예시: 일 매출 1억원 → 시간당 약 417만원 손실

사용자 생산성 손실 = 영향받는 인원 × 시급 × 중단 시간
예시: 100명 × 5만원 × 2시간 = 1,000만원 손실
```

**정성적 영향 요소**:
- 고객 만족도 저하
- 브랜드 이미지 손상
- 규정 준수 위반 리스크
- 경쟁사 대비 우위 상실

### 비즈니스 연속성 관점

**핵심 비즈니스 프로세스별 RTO/RPO**:

| 비즈니스 프로세스 | RTO (복구 목표 시간) | RPO (복구 목표 시점) | 최대 허용 중단 |
|-------------------|---------------------|---------------------|---------------|
| **온라인 주문 처리** | 15분 | 0분 | 15분 |
| **고객 서비스** | 30분 | 5분 | 30분 |
| **재고 관리** | 2시간 | 15분 | 2시간 |
| **회계 처리** | 4시간 | 1시간 | 1일 |

## 에스컬레이션 기준 및 절차

### 자동 에스컬레이션 규칙

**시간 기반 에스컬레이션**:

| 우선순위 | 1차 에스컬레이션 | 2차 에스컬레이션 | 3차 에스컬레이션 |
|----------|-----------------|-----------------|-----------------|
| **P1** | 15분 후 팀장 | 30분 후 부서장 | 1시간 후 임원 |
| **P2** | 1시간 후 팀장 | 4시간 후 부서장 | 1일 후 임원 |
| **P3** | 4시간 후 팀장 | 1일 후 부서장 | 3일 후 임원 |
| **P4** | 1일 후 팀장 | 3일 후 부서장 | 1주 후 임원 |

**상황 기반 에스컬레이션**:
- **미디어 관심사**: 즉시 PR팀 및 경영진 통보
- **고객 클레임**: 고객서비스팀 및 영업팀 연계
- **보안 이슈**: 보안팀 및 법무팀 즉시 참여
- **규정 위반**: 컴플라이언스팀 및 감사팀 통보

### 에스컬레이션 프로세스

**1단계: 기술적 에스컬레이션**
1. 1차 지원팀에서 해결 불가 시
2. 2차 전문팀으로 이관
3. 필요시 외부 벤더 지원 요청

**2단계: 관리적 에스컬레이션**
1. 목표 해결 시간 초과 시
2. 추가 리소스 또는 권한 필요 시
3. 비즈니스 의사결정 필요 시

**3단계: 경영진 에스컬레이션**
1. 전사적 영향 발생 시
2. 고객 또는 언론 이슈화 시
3. 외부 기관 보고 필요 시

## 실무 활용 예시

### 상황 1: 전자상거래 사이트 결제 시스템 오류

**목표**: 매출 직결 시스템의 신속한 우선순위 결정과 대응

**분류 및 우선순위 결정**:
1. **서비스 분류**: Tier 1 (Critical) - 고객 대면 매출 서비스
2. **기술적 분류**: Application - 결제 시스템
3. **영향도**: 높음 (전체 고객 결제 불가)
4. **긴급도**: 높음 (업무시간 중 매출 손실)
5. **최종 우선순위**: P1 (Critical)

**즉시 조치**:
- 15분 내 전담팀 구성
- 결제 대행사 및 은행 연락
- 고객 공지 및 대체 결제 수단 안내
- 경영진 즉시 보고

**결과**: 30분 내 임시 복구, 1시간 내 완전 복구 달성

### 상황 2: 특정 부서 네트워크 프린터 오류

**목표**: 제한적 영향의 적절한 우선순위 설정으로 리소스 효율 활용

**분류 및 우선순위 결정**:
1. **서비스 분류**: Tier 3 (Standard) - 일반 업무 지원 도구
2. **기술적 분류**: End-User - 프린터/주변기기
3. **영향도**: 낮음 (5명 사용자, 대체 프린터 존재)
4. **긴급도**: 낮음 (워크어라운드 용이)
5. **최종 우선순위**: P4 (Low)

**대응 방법**:
- 임시 해결방안 즉시 제공
- 정기 점검 시간에 근본 수리
- 예방 정비 계획에 포함

**결과**: 즉시 대체 방안 제공, 다음날 완전 수리 완료

### 상황 3: 개발 서버 하드웨어 장애

**목표**: 개발 환경 장애의 비즈니스 우선순위 고려한 효율적 대응

**분류 및 우선순위 결정**:
1. **서비스 분류**: Tier 4 (Low) - 개발/테스트 환경
2. **기술적 분류**: Server - 물리 서버 하드웨어
3. **영향도**: 중간 (개발팀 10명 영향, 릴리스 일정 지연 가능)
4. **긴급도**: 중간 (프로젝트 마일스톤 고려)
5. **최종 우선순위**: P3 (Medium)

**특별 고려사항**:
- 다음 주 중요 릴리스 예정으로 긴급도 상향 조정
- 개발팀장과 협의하여 P2로 재분류
- 백업 개발 환경 즉시 제공

**결과**: 4시간 내 백업 환경 제공, 24시간 내 완전 복구

## 자동화 및 AI 활용

### 지능형 분류 시스템

**머신러닝 기반 자동 분류**:
```python
분류 알고리즘 요소:
- 키워드 가중치 분석
- 과거 유사 사례 패턴 학습
- 신고자 위치/부서 정보 활용
- 시간대별 비즈니스 영향 고려

정확도 목표:
- 1차 자동 분류: 85% 정확도
- 2차 수동 검토: 98% 정확도
```

**동적 우선순위 조정**:
- 실시간 비즈니스 상황 반영
- 사용자 수 및 영향도 자동 계산
- 외부 요인 (이벤트, 공휴일) 고려
- SLA 마감 시간 자동 추적

:::success 자동화 효과
지능형 분류 시스템 도입 후 수동 분류 작업 70% 감소, 분류 정확도 15% 향상 달성
:::

### 예측적 우선순위 관리

**트렌드 분석 기반 예측**:
- 계절성 패턴 분석 (시즌별 트래픽 증가)
- 주기적 이벤트 영향 (정기 점검, 업데이트)
- 외부 환경 요인 (날씨, 사회적 이슈)

## 성과 측정 및 개선

### 분류 정확도 지표

| 지표명 | 목표 수치 | 측정 방법 | 현재 달성도 |
|--------|-----------|-----------|-------------|
| **분류 정확도** | > 95% | 올바른 분류 / 전체 분류 | 92% |
| **우선순위 적절성** | > 90% | 적절한 우선순위 / 전체 | 88% |
| **에스컬레이션 적정성** | < 5% | 불필요한 에스컬레이션 비율 | 7% |
| **재분류 발생률** | < 10% | 재분류된 인시던트 비율 | 12% |

### 지속적 개선 활동

**주간 검토 프로세스**:
1. **분류 오류 분석**
   - 잘못 분류된 사례 상세 분석
   - 분류 규칙 개선점 도출
   - 교육 필요 영역 식별

2. **우선순위 적절성 평가**
   - 비즈니스 영향과 설정 우선순위 비교
   - 에스컬레이션 발생 원인 분석
   - 임계값 조정 필요성 검토

**월간 최적화 활동**:
- 분류 체계 업데이트
- 자동화 규칙 튜닝
- 교육 프로그램 실시
- 성과 지표 목표 재검토

## 문제 해결 가이드

### 자주 발생하는 문제들

#### 문제: 중요한 인시던트가 낮은 우선순위로 분류됨

**원인**: 비즈니스 영향 평가 기준의 부정확성 또는 자동 분류 규칙 한계

**해결**:
1. **비즈니스 영향 재평가**
   - 실제 영향받는 사용자 수 재확인
   - 비즈니스 프로세스 중단 범위 재평가
   - 매출 영향 및 고객 영향 재계산

2. **분류 규칙 개선**
   - 핵심 키워드 가중치 조정
   - 시스템별 중요도 매트릭스 업데이트
   - 예외 상황 처리 규칙 추가

#### 문제: 에스컬레이션이 너무 빈번하게 발생함

**원인**: 에스컬레이션 임계값 설정이 부적절하거나 1차 대응 역량 부족

**해결**:
1. **임계값 재조정**
   - 과거 해결 시간 분석을 통한 현실적 목표 설정
   - 우선순위별 차등화된 임계값 적용
   - 업무 시간과 비업무 시간 구분 적용

2. **1차 대응 역량 강화**
   - 담당자 기술 교육 강화
   - 해결 가이드 및 매뉴얼 개선
   - 도구 및 권한 확대 검토

:::warning 분류 및 우선순위 설정 시 주의사항
- **일관성 유지**: 동일한 상황에 대해 항상 같은 기준 적용
- **문서화 철저**: 모든 분류 및 우선순위 결정 근거 명확히 기록
- **정기적 교정**: 비즈니스 환경 변화에 따른 기준 주기적 업데이트
:::
