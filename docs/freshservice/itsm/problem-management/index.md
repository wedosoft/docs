# 문제 관리

문제 관리는 인시던트의 수와 심각도를 최소화하기 위해 근본 원인을 체계적으로 찾고 해결하는 ITIL 4 기반의 핵심 프로세스입니다.

:::info 문제 관리 구현 전 고려사항
- **인시던트 관리와의 연계**: 반복되는 인시던트 패턴을 분석하여 근본적인 해결책 제시
- **예방적 접근**: 미래 인시던트 발생을 방지하는 사전 대응 체계 구축
- **Known Error Database**: 체계적인 지식 관리를 통한 문제 해결 효율성 향상
:::

## 문제 관리 프로세스 개요

### Reactive Problem Management (반응적 문제 관리)
발생한 인시던트를 기반으로 문제를 식별하고 해결하는 접근법입니다.

**주요 활동:**
1. **문제 식별**: 반복되는 인시던트 패턴 분석
2. **문제 로깅**: 공식적인 문제 레코드 생성
3. **조사 및 진단**: 근본 원인 파악을 위한 체계적 분석
4. **해결책 구현**: 영구 해결책 개발 및 적용
5. **문제 종료**: 해결 검증 및 Known Error 등록

### Proactive Problem Management (사전적 문제 관리)
잠재적 문제를 사전에 식별하고 예방하는 접근법입니다.

**주요 활동:**
- **트렌드 분석**: 시스템 성능 및 인시던트 데이터 분석
- **위험 평가**: 잠재적 장애 요소 식별
- **예방 조치**: 문제 발생 전 사전 대응
- **지속적 개선**: 서비스 품질 향상을 위한 개선 활동

## 문제 분류 및 우선순위

### 문제 분류 기준

| 분류 | 기준 | 예시 |
|------|------|------|
| **Critical** | 전사 서비스 중단 | 코어 시스템 장애, 네트워크 전면 중단 |
| **High** | 주요 서비스 영향 | 특정 시스템 장애, 성능 심각 저하 |
| **Medium** | 부분적 서비스 영향 | 특정 기능 오류, 일부 사용자 영향 |
| **Low** | 최소한의 영향 | 비핵심 기능 문제, 개별 사용자 이슈 |

### 우선순위 결정 매트릭스

```markdown
우선순위 = (영향도 × 3) + (빈도 × 2) + (긴급도 × 1)

영향도 점수:
- 5: 전사 비즈니스 중단
- 4: 주요 부서 업무 중단  
- 3: 특정 기능 중단
- 2: 성능 저하
- 1: 경미한 불편

빈도 점수:
- 5: 일일 반복 발생
- 4: 주간 반복 발생
- 3: 월간 반복 발생
- 2: 분기별 발생
- 1: 연간 1-2회 발생
```

## Known Error 관리

### Known Error 등록 기준

**필수 조건:**
- ✅ 근본 원인이 명확히 파악된 문제
- ✅ 검증된 해결책 또는 임시 해결책 보유
- ✅ 재발 가능성이 있는 문제
- ✅ 문서화 가능한 수준의 상세 정보

### Known Error Database (KEDB) 구조

```markdown
📋 KEDB 필수 정보:
- Error ID: [KERR-YYYY-NNNN]
- 문제 요약: [간결한 문제 설명]
- 증상: [사용자가 경험하는 현상]
- 근본 원인: [기술적 원인 분석]
- 임시 해결책: [즉시 적용 가능한 Workaround]
- 영구 해결책: [완전한 문제 해결 방법]
- 관련 인시던트: [연관된 인시던트 목록]
- 업데이트 이력: [변경 사항 기록]
```

## 문제 관리 성과 지표

### 핵심 KPI

| 지표명 | 목표 수치 | 측정 방법 | 설명 |
|--------|-----------|-----------|------|
| **Problem Resolution Rate** | > 80% | 해결된 문제 / 전체 문제 | 문제 해결률 |
| **Recurring Incident Reduction** | &lt; 10% | 반복 인시던트 발생률 | 재발 방지 효과 |
| **Known Error Utilization** | > 60% | Known Error 활용 해결률 | 지식 활용도 |
| **Mean Time to Resolution** | &lt; 30일 | 평균 문제 해결 시간 | 해결 효율성 |
| **Prevention Effectiveness** | > 70% | 예방된 인시던트 추정 수 | 사전 관리 효과 |

### 모니터링 대시보드

```markdown
실시간 문제 관리 현황:
🔍 활성 문제: 12건
📋 조사 중: 8건
⚠️ 해결 대기: 3건
✅ 해결 완료: 145건 (월간)
📚 Known Error: 89건
```

## 실무 활용 예시

### 상황 1: 반복적인 이메일 서버 성능 저하
**목표**: 주간 2-3회 발생하는 이메일 지연 문제의 근본 원인 파악 및 해결

**문제 관리 프로세스 적용:**

1. **문제 식별 및 로깅**
   - 인시던트 패턴 분석: 매주 화요일, 목요일 오전 이메일 지연
   - 문제 레코드 생성: PRB-2024-0156
   - 영향도 평가: High (전체 사용자 1,000명 영향)

2. **조사 및 진단**
   - 이메일 서버 로그 분석
   - 네트워크 트래픽 모니터링
   - 백업 작업 스케줄 검토
   - 데이터베이스 성능 분석

3. **근본 원인 발견**
   - 매주 화/목요일 오전 자동 백업과 바이러스 스캔이 동시 실행
   - 디스크 I/O 포화로 인한 이메일 서비스 성능 저하
   - 메모리 부족으로 인한 큐잉 지연

4. **해결책 구현**
   - **임시 해결책**: 백업 시간을 야간으로 임시 변경
   - **영구 해결책**: 백업 스케줄 재설계, 하드웨어 업그레이드
   - **변경 관리**: CAB 승인을 통한 체계적 구현

**결과**: 이메일 성능 문제 100% 해결, 유사 문제 재발 방지

:::success 문제 해결 성공 요인
- 체계적인 데이터 수집과 패턴 분석
- 다각도 원인 분석을 통한 정확한 진단
- 임시와 영구 해결책의 단계적 적용
:::

### 상황 2: 사용자 계정 잠금 문제의 사전적 관리
**목표**: 계정 잠금 인시던트 급증 트렌드의 사전 대응

**Proactive Problem Management 적용:**

1. **트렌드 분석**
   - 최근 3개월간 계정 잠금 인시던트 300% 증가
   - 특정 부서(영업팀) 집중 발생
   - 새로운 CRM 시스템 도입 시점과 일치

2. **근본 원인 분석**
   - CRM과 AD 인증 연동 이슈 발견
   - 사용자 교육 부족으로 인한 잘못된 로그인 시도
   - 자동 로그인 정책의 복잡성

3. **예방 조치 구현**
   - CRM-AD 연동 로직 개선
   - 사용자 대상 인증 정책 교육 실시
   - 계정 잠금 임계값 조정

**결과**: 계정 잠금 인시던트 70% 감소, 사용자 만족도 향상

### 상황 3: Known Error를 활용한 신속 대응
**목표**: 기존 Known Error를 활용한 효율적 인시던트 해결

**시나리오**: 주말 야간 데이터베이스 연결 오류 발생

1. **Known Error 검색**
   - 증상 기반 KEDB 검색: "데이터베이스 연결 타임아웃"
   - 매칭되는 Known Error 발견: KERR-2024-0089

2. **즉시 대응**
   - Known Error의 임시 해결책 적용
   - 연결 풀 설정 조정 및 서비스 재시작
   - 15분 내 서비스 정상화

3. **사후 조치**
   - 영구 해결책 적용 일정 수립
   - Known Error 정보 업데이트
   - 유사 시스템 예방 점검 실시

**결과**: 서비스 중단 시간 최소화, 근무 외 시간 신속 대응

## 문제 관리 도구 및 기법

### 근본 원인 분석 기법

**5 Why 분석**
```markdown
예시: 웹 서비스 응답 지연
Why 1: 왜 응답이 느린가? → 데이터베이스 쿼리가 느림
Why 2: 왜 쿼리가 느린가? → 인덱스가 없음
Why 3: 왜 인덱스가 없는가? → 설계 시 고려되지 않음
Why 4: 왜 설계 시 고려되지 않았나? → 성능 요구사항 명시 부족
Why 5: 왜 요구사항이 명시되지 않았나? → 요구사항 분석 프로세스 미흡
```

**Fishbone Diagram (특성요인도)**
- **인적 요소**: 교육 부족, 경험 부족, 의사소통 오류
- **프로세스**: 절차 미비, 체크리스트 부재, 승인 지연
- **도구/기술**: 장비 노후, 소프트웨어 버그, 호환성 문제
- **환경**: 네트워크 상태, 전력 문제, 물리적 조건

:::tip 효과적인 문제 관리를 위한 핵심 원칙
- **체계적 접근**: 감정이 아닌 데이터와 사실에 기반한 분석
- **협업 중심**: 다부서 전문가들의 통합적 시각 활용
- **지속적 개선**: 해결된 문제도 정기적 검토를 통한 개선
- **지식 공유**: Known Error Database를 통한 조직 학습 촉진
:::

효과적인 문제 관리를 통해 조직은 서비스 안정성을 크게 향상시키고, 반복적인 인시던트로 인한 비용과 업무 중단을 최소화할 수 있습니다.
